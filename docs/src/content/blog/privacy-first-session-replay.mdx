Session replay is one of the most useful debugging tools a product team has, and one of the easiest to misuse. The same recording that helps you reproduce a checkout bug can also capture a customer's full name, their email address, the contents of a private message they were halfway through writing, and a profile photo of their face. Most replay tools let you mask all of that — *if you remember to configure it*. We didn't want "if you remember" in the design.

This post walks through what Betterlytics's [session replay](/docs/dashboard/session-replay) does by default, why we picked those defaults, and what they cost you in debugging power. The short version: every text node is replaced with asterisks before it leaves the browser, every `<img>` is blocked entirely, every `<iframe>` is blocked, and there is no opt-out for "just this one input." If you want to record something readable, you mark it readable explicitly. The dangerous direction is the configured one.

## What "by default" actually means

Most teams who turn on session replay accept the vendor's default privacy posture without auditing it. So the default has to be the version you'd want a regulator to look at first. Here's exactly what Betterlytics's tracker passes to [rrweb](https://github.com/rrweb-io/rrweb) when recording is enabled:

- **`maskAllInputs: true`** — every input element gets its value masked, regardless of type. Passwords, emails, phone numbers, search boxes, textareas, dropdowns. There is no input type that ships unmasked.
- **`maskTextSelector: "*"`** — every text node on the page is treated as sensitive. The default rrweb behaviour is to mask only inputs and elements you tag; we widen the selector to the universal one. A nine-character string of text becomes `*********` in the recording.
- **`blockSelector: "img, iframe, ..."`** — `<img>` and `<iframe>` elements are blocked outright. The replay shows a placeholder where the image was, not the image itself. We also block Recharts SVG content, because dashboards built on top of Betterlytics often render customer data inside charts.
- **`slimDOMOptions`** — scripts, comments, favicon links, and most meta tags are stripped from the captured DOM. They aren't useful for replay and they're a common place for tracking parameters and identifiers to leak.

The asterisk substitution preserves length but not content. "Hi John Doe" becomes `** **** ***` — you can see *that* text was there, and roughly how much of it, but you cannot read it. That's enough to debug "this button overlaps the username on mobile" and not enough to leak a username.

## Why block images instead of masking them

Masking text is a solved problem — replace characters, preserve layout. Masking images is harder. The honest options are:

1. **Inline the image, blur it server-side.** Requires your replay backend to process every frame. Expensive, brittle, and you've still transmitted the original to the server.
2. **Inline the image, blur it client-side.** Works for `<img>` with same-origin sources. Doesn't work for cross-origin images, CSS background images, or canvas-rendered avatars.
3. **Block the element entirely.** The replay shows a placeholder rectangle where the image was. You lose visual fidelity. You gain a guarantee.

We picked option 3. A blocked image is a deliberately worse debugging experience — you can't see whether the avatar rendered correctly — but the worst case is "we couldn't reproduce a visual bug," not "we accidentally stored a customer's face for 60 days." For the cases where you genuinely need an image visible (a logo, a marketing illustration), you can opt it back in by adding the `rr-ignore` class to the element. Opting *in* to recording is a more honest design than opting *out* of leakage.

## The tradeoffs we accept

This default is strict enough that it costs us things real teams ask for. We think they're worth losing.

- **You can't read error messages on the page.** If your app shows "Couldn't save: invalid email", the recording shows `********: ******* *****`. You see that an error appeared, but not what it said. This is fine — the error itself should be in your error tracker, with the exact message and stack. Replay's job is the *interaction*, not the message text.
- **Your own marketing copy is masked.** The hero headline on your landing page, the CTA button text, your own pricing table — all asterisks. We considered allowing self-owned content through, but "self-owned" is impossible to detect from the DOM. So everything goes.
- **Dashboards rendered inside your app become unreadable.** If your product itself shows tables of customer data, the replay won't show those values. Again — fine. The values are in your database; you don't need them in the recording.
- **Recharts content is blocked.** This is specific: if your app uses [Recharts](https://recharts.org) (a common React charting library), the rendered SVG often contains real customer data labels. We block those SVGs in the recording for the same reason we block images.

The one place we do *more* than a blanket asterisk is for elements explicitly marked sensitive — `[data-rrweb-mask]`, `.rr-block`, `.rr-ignore`. Those are still hidden. A sensitive thing can't accidentally become visible just because you removed the universal masking; the explicit markers always win.

## What this means for the regulators

Betterlytics is built around the position that you should not need a cookie banner for analytics. Session replay is a separate question — it is more invasive than pageview counting, and most EU data protection authorities treat it as something that needs explicit consent. The CNIL has [published guidance](https://www.cnil.fr/en/cookies-and-other-tracking-devices-cnil-publishes-new-guidelines) treating behavioural recording as a higher bar than basic measurement, and we agree.

So Betterlytics's session replay ships **disabled by default**. Enabling it is a deliberate `data-replay="true"` decision, and the tracker exposes `window.betterlytics.setReplayConsent(true)` so you can gate it behind your existing consent manager. The masking defaults are not a substitute for consent — they are the second line of defence for the recordings you are legally allowed to make.

We also retain replays for 60 days and then delete them irreversibly. Combined with the masking and blocking defaults, the worst-case data exposure for a single user is "60 days of asterisks and placeholder rectangles." That feels like the right ceiling for a feature whose primary purpose is debugging.

## Try it, then look at it

If you already use Betterlytics, enable replay on a staging site with `data-replay="true"`, generate a few sessions, and then *watch the replays as if you were a regulator*. The test we use internally: pause the player on a screen that should contain a real person's name. If you can read the name, the masking has failed. If you see asterisks, the defaults did their job.

If you don't use Betterlytics yet, our [session replay integration guide](/docs/integration/session-replay) walks through the script tag and the npm package, and the [dashboard guide](/docs/dashboard/session-replay) covers the player. Both fit on a single page, because the configuration surface is small on purpose.
